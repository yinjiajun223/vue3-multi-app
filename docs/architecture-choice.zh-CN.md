# 架构选型：为什么当前模板使用多入口，而不是 Monorepo + Turborepo

## 结论先行

对于这个项目模板的目标（同一仓库下的前台 + 后台、共享代码较多、快速启动、低运维复杂度），`Vite 多入口`比 `Monorepo + Turborepo` 更合适。

## 为什么这里优先选择多入口

### 1) 复杂度更低，上手更快

- 多入口只需要一个 `package.json`、一套依赖和一套构建配置。
- 新同学拉代码后，`npm install && npm run dev` 即可启动。
- 对模板项目来说，心智负担明显小于 workspace + task pipeline。

### 2) 更适合“强耦合”的前后台

- 前台和后台通常共享同一套组件、请求封装、工具函数和状态约定。
- 在多入口下，共享代码天然在同一个 `src` 树内，复用路径最短。
- 避免了跨包依赖和版本同步的额外成本。

### 3) 本地开发体验直接

- 一个端口同时调试两个入口：
  - `/`（web）
  - `/admin.html`（admin）
- 无需额外启动 orchestrator，也不需要学习任务拓扑。

### 4) CI/CD 成本更可控

- 单项目流水线即可覆盖 lint/type-check/build。
- 对中小团队和模板项目，维护成本更低。

## 什么时候应该考虑 Monorepo + Turborepo

当你的项目出现以下特征时，Monorepo 方案会更有优势：

- 仓库内有多个相对独立应用（例如 web、admin、h5、docs、cli）。
- 有可独立发布的共享包（如 `@company/ui`、`@company/eslint-config`、`@company/sdk`）。
- 需要大规模增量构建、任务缓存、跨包并行流水线优化。
- 团队规模和仓库规模已经使单工程方案变得笨重。

## 选型建议（实操）

- 当前模板阶段：继续使用 `多入口单工程`，保证简单和稳定。
- 后续演进阈值：当“独立应用数 ≥ 3 且存在独立发布包”时，再迁移到 Monorepo + Turborepo。

## 本模板当前定位

- 目标：提供“可复制、可快速起步”的企业常见前后台模板。
- 关键能力：
  - 双入口路由
  - 统一 axios 封装
  - 共享组件与工具
  - 统一工程规范（ESLint、Commitlint、Husky、lint-staged）

以上目标与多入口方案高度匹配，因此当前选择是合理且务实的。
